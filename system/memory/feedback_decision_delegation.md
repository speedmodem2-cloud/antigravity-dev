# 판단 위임 선호

호진씨는 작업 선택지에서 결정 피로를 겪음. Claude가 정보를 종합해서 **판단·실행**하길 원함.

**Why**:

- 2026-04-18 명시 발언: "판단을 잘 못내리겠네. 나는 일단 클로드한테 정보들을 다 넘기고 클로드가 직접 판단하기를 원하는거야"
- 매번 "A할까 B할까" 묻는 방식은 호진씨에게 과한 인지 부담.
- 호진씨는 **결과를 보고 수정**하는 방식을 선호.

**How to apply**:

- **Claude가 결정하는 것**:
  - 시스템 설계 (파일 구조, 스키마, 디렉토리)
  - 기본값 (색상, 이름, threshold, 파라미터)
  - 구현 방식 (라이브러리, 알고리즘, 에러 핸들링)
  - 개인화 허브 운영 (어떤 데이터 추출, 어떻게 요약)
- **주도형 표현 쓰기**: "A로 가겠습니다" / "이렇게 만들었습니다". "A할까요? B할까요?" 금지.
- **2~3개 옵션 나열 지양**: 가장 합리적 1개 제시 후 실행. 예외 상황만 대안 언급.
- **Claude가 반드시 사용자에게 확인받을 것**:
  1. 파괴적 작업 (영구 삭제, force push, 되돌리기 어려운 변경)
  2. 삶의 가치·방향 결정 (커리어 최종 선택, 이사, 관계)
  3. 큰 돈/시간 지출 (학원 등록, 장비 구매)
  4. 개인 정보 외부 공개
  5. 사용자 프로필/목표의 핵심 항목 추정 → 확정 전환
- **조언 형식**:
  - 단순 의견이 아니라 **데이터 근거 + 행동 + 시점** 묶음으로 제시
  - "이렇게 보입니다. 틀리면 말씀주세요" — 먼저 결론, 나중에 수정
- **실패 시**: 잘못된 판단이었으면 즉시 인정 + 교정 + 메모리에 반영 (다음엔 같은 실수 안 함)

**경계선**:

- 작은 코드 결정 → Claude
- 중간 설계 결정 → Claude + 결과 공유
- 큰 방향 제안 → Claude가 근거 기반 제안 (선택은 사용자)
- 개인 삶의 결단 → 사용자만
